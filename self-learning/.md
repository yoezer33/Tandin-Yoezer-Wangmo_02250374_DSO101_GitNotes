# Beginner Notes — Render.com, Linux & GitHub Actions

---

# 1. Render.com

Render.com is a cloud platform that lets you host your web applications on the internet without managing any servers yourself. Instead of setting up your own machine, you simply connect your GitHub account to Render, and every time you push new code, Render automatically builds and deploys your app for you. It is one of the easiest ways to get a project live online.

## What Render Can Host

Render supports several types of services depending on what your project needs. A **Web Service** is used for backend applications built with Node.js, Python, or any other language that listens on a port. A **Static Site** is for frontend apps like React or plain HTML and CSS that do not need a backend server. Render also offers a managed **PostgreSQL** database, which means Render takes care of running and maintaining the database for you. Finally, if you have already built a **Docker image** and pushed it to Docker Hub, Render can pull and deploy that image directly.

## How to Deploy on Render

The first thing you do is sign up at render.com and connect your GitHub account so Render can access your repositories. Once connected, you click **New → Web Service** and either select a GitHub repo or provide a Docker Hub image name. Before deploying, you must add your environment variables — things like `DB_HOST`, `DB_PASSWORD`, and `PORT` — under the Environment tab in the dashboard. These are secret settings that your app needs to run, and they should never be written directly inside your code. After setting them, you click **Deploy** and Render builds and starts your application. From that point on, every future push to your `main` branch triggers a fresh deployment automatically. Render will give you a live URL in the format `https://your-app.onrender.com` where anyone can access your app.

## render.yaml — Deploying Multiple Services at Once

When your project has both a frontend and a backend, you can create a file called `render.yaml` in the root of your repository. This file tells Render about all your services in one place, so you do not have to configure them manually one by one. Render reads this file when you connect your repo and sets everything up automatically. Every new Git push then rebuilds and redeploys all services at once with zero manual steps.

```yaml
services:
  - type: web
    name: be-todo
    env: docker
    dockerfilePath: ./backend/Dockerfile
    envVars:
      - key: DB_HOST
        value: your-db-host
      - key: PORT
        value: 5000

  - type: web
    name: fe-todo
    env: docker
    dockerfilePath: ./frontend/Dockerfile
    envVars:
      - key: REACT_APP_API_URL
        value: https://be-todo.onrender.com
```

## Key Things to Know

A **Deploy Hook URL** is a special link that Render gives you. When something sends an HTTP request to that link — like a GitHub Actions workflow — Render immediately starts a new deployment. This is how CI/CD pipelines trigger Render without you clicking anything manually. One important thing to know about the **free tier** is that apps go to sleep after 15 minutes of inactivity. The next time someone visits, the app takes about 30 seconds to wake up. This is normal behaviour on the free plan.

# 2. Linux Basics

Linux is an operating system just like Windows or macOS, but it is used by almost every server and cloud platform in the world, including Render, Docker, and GitHub Actions runners. The key difference from Windows is that you control Linux entirely by typing commands into a **terminal** — there are no windows to click or drag. At first this feels unfamiliar, but after learning a handful of commands you can do almost anything.

## Moving Around the File System

When you open a terminal, you are always standing somewhere inside the file system, just like being inside a folder on your desktop. The command `pwd` (print working directory) tells you exactly where you are right now. To see what files and folders exist in your current location, you type `ls`. If you want to see hidden files as well (files whose names start with a dot), you use `ls -la`, which also shows file sizes and permissions. To move into a folder, you type `cd foldername`. To go back up one level to the parent folder, you type `cd ..`. To jump straight to your home directory from anywhere, you type `cd ~`.

## Creating, Copying, and Deleting Files

To create a new empty folder, you use `mkdir` followed by the name you want. To create a new empty file, you use `touch filename.txt`. If you want to read what is inside a file without opening an editor, `cat filename.txt` prints the entire contents directly in the terminal. Copying a file is done with `cp original.txt copy.txt`, and renaming or moving a file uses `mv oldname.txt newname.txt`. Deleting a single file uses `rm filename.txt`. To delete an entire folder along with everything inside it, you use `rm -rf foldername/` — be very careful with this command because the deletion is permanent and there is no recycle bin on Linux.

## File Permissions

Every file on Linux has permissions that control who is allowed to read it, write to it, or run it as a program. When you get a `permission denied` error, it usually means the file is not set as executable. The command `chmod +x file.sh` adds execute permission so the file can be run as a script. A more specific form is `chmod 755 file`, which means the owner can read, write, and run it, while everyone else can only read and run it. When you need to run a command as an administrator (for example, installing software), you put `sudo` in front of it. Use `sudo` carefully because it has full power over the system.

## Processes and Useful System Commands

A process is any program that is currently running on the system. The command `ps aux` lists all running processes along with their ID numbers. If a process is frozen or causing problems, you can stop it with `kill PID`, where PID is the process ID number shown by `ps aux`. The `top` command gives you a live, updating view of all processes and how much CPU and memory they are using — press Q to exit it. The `echo` command simply prints text to the terminal, which is useful inside scripts. When you are done with a terminal session or want to exit a container shell, you type `exit`. To clear all the previous output from the screen, you type `clear`.

## Installing Software

On Ubuntu and Debian-based Linux systems (which is what most cloud servers use), you install software using the `apt` package manager. Before installing anything, you should run `apt update` to refresh the list of available packages. Then you install a specific tool with `apt install toolname`, for example `apt install curl`.

```bash
apt update          # refresh the list of available packages
apt install curl    # install a package
```


# 3. GitHub Actions

GitHub Actions is an automation tool that is built directly into GitHub. Its purpose is to automatically run a series of tasks every time something happens in your repository — most commonly when you push new code. These tasks can include installing dependencies, running tests, building Docker images, pushing those images to Docker Hub, and deploying your app to Render. The big advantage is that you do not have to do any of these steps manually — they all happen on their own.

## How GitHub Actions Works

The entire process starts when you push code to your repository. GitHub detects the push and looks for a special file inside your repo at the path `.github/workflows/deploy.yml`. This file, written in YAML format, contains all the instructions for what GitHub should do. GitHub then creates a temporary virtual machine called a **runner** — essentially a fresh Linux computer in the cloud — and runs your instructions on it. The instructions are organised into **jobs**, and each job contains a list of **steps** that execute one after another. Once all steps finish, you can go to the **Actions** tab in your GitHub repository and see a green checkmark for success or a red X for failure. Clicking on any run shows you the full logs for every step, which makes it easy to debug problems.

## The Workflow File Explained

The workflow file lives at `.github/workflows/deploy.yml` and uses YAML syntax. At the top, `name:` gives your workflow a label that appears in the Actions tab. The `on:` section defines the trigger — in most projects this is `push: branches: [main]`, meaning the workflow runs every time code is pushed to the main branch. Inside `jobs:`, you define one or more jobs. Each job has a `runs-on:` field that picks the operating system for the runner — `ubuntu-latest` is the most common choice. Then inside `steps:`, each step either runs a shell command using `run:` or uses a pre-built action from the GitHub Marketplace using `uses:`. The most common pre-built action is `actions/checkout@v3`, which simply clones your repository onto the runner so the rest of your steps can access your code.

```yaml
# .github/workflows/deploy.yml

name: Build and Deploy

on:
  push:
    branches: [ main ]       # runs when you push to main

jobs:
  deploy:
    runs-on: ubuntu-latest   # use a Linux machine

    steps:
      - name: Checkout code
        uses: actions/checkout@v3

      - name: Login to Docker Hub
        run: echo "${{ secrets.DOCKERHUB_TOKEN }}" | docker login -u "${{ secrets.DOCKERHUB_USERNAME }}" --password-stdin

      - name: Build and push image
        run: |
          docker build -t myuser/myapp:latest .
          docker push myuser/myapp:latest

      - name: Deploy to Render
        run: curl "${{ secrets.RENDER_DEPLOY_HOOK_URL }}"
```

## GitHub Secrets — Keeping Passwords Safe

One of the most important things to understand about GitHub Actions is how to handle passwords and tokens safely. You should never write a password, API key, or token directly inside your workflow file, because anyone who can see your repository can read that file. Instead, GitHub provides a feature called **Secrets**. You go to your repository's **Settings → Secrets and variables → Actions** and add your sensitive values there. In the workflow file, you reference them using the syntax `${{ secrets.SECRET_NAME }}`. GitHub replaces that placeholder with the actual value at runtime, and the value is never shown in the logs.

In this project, three secrets were used. `DOCKERHUB_USERNAME` holds the Docker Hub account name so the pipeline knows which account to push images to. `DOCKERHUB_TOKEN` is a special access token generated from Docker Hub — it is used instead of a real password so that the actual password is never exposed anywhere. `RENDER_DEPLOY_HOOK_URL` is the webhook URL from Render.com that, when called with `curl`, tells Render to start a new deployment with the latest image.

## Key Terms Summary

A workflow is the entire automation process defined in one `.yml` file. The trigger  defined under `on:` is what starts the workflow, such as a push to main. A job is a group of steps that all run on the same machine. A step is one individual command or action inside a job. The runneris the temporary virtual machine that GitHub provides to execute the job. Secrets are encrypted values stored in GitHub Settings that let you use sensitive data without exposing it in your code.

> **Tip:** If a workflow fails, go to the **Actions** tab, click the failed run, then click the failed step to expand its log output. The error message will tell you exactly what went wrong.
